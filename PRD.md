## 1. 产品概述

### 1.1 产品定位

Dating IN BNBU 是一款面向 **北京师范大学 - 香港浸会大学联合国际学院 (UIC)** 学生的校园交友配对平台。通过基于兴趣、目标和学术背景的智能化匹配算法，帮助用户找到志同道合的伙伴。

### 1.2 产品愿景

打造一个安全、真实、高效的校园社交环境，让学生能够：
- 找到学习兴趣相投的伙伴
- 发现目标一致的同行者
- 拓展校园人脉圈

### 1.3 核心价值主张

| 价值点 | 说明 |
|--------|------|
| 🎓 **校园专属** | 仅限 UIC 学生和教职工，需使用校园邮箱注册 |
| 🎯 **精准匹配** | 基于学术目标、兴趣爱好、GPA 等多维度智能打分 |
| 🔒 **安全真实** | JWT 认证 + BCrypt 密码加密，保障用户信息安全 |
| 📱 **移动优先** | React Native 跨平台应用，iOS/Android 无缝体验 |

├─────────────────────────────────────────┤
│ id (PK)                                 │

│ match_score     │                       │

└─────────────────────────────────────────┘

### 6.2 表结构详情

#### user - 用户账号表

```sql
CREATE TABLE `user` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '用户 ID',
    `edu_email` VARCHAR(128) NOT NULL UNIQUE COMMENT '校园邮箱',
    `password_hash` VARCHAR(255) NOT NULL COMMENT 'BCrypt 密码哈希',
    `status` VARCHAR(32) DEFAULT 'ACTIVE' COMMENT '账户状态',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户账号表';
```

#### user_profile - 用户画像表

```sql
CREATE TABLE `user_profile` (
    `user_id` BIGINT PRIMARY KEY COMMENT '用户 ID(外键)',
    `nickname` VARCHAR(64) COMMENT '昵称',
    `major` VARCHAR(128) COMMENT '专业',
    `gpa_level` VARCHAR(64) COMMENT 'GPA 区间',
    `hobbies` TEXT COMMENT '兴趣爱好(JSON 数组)',
    `goals` TEXT COMMENT '阶段目标(JSON 数组)',
    `match_intent` VARCHAR(255) COMMENT '匹配意向',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (`user_id`) REFERENCES `user`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户画像表';
```

#### match_record - 匹配记录表

```sql
CREATE TABLE `match_record` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '记录 ID',
    `user_id1` BIGINT NOT NULL COMMENT '用户 1 ID',
    `user_id2` BIGINT NOT NULL COMMENT '用户 2 ID',
    `match_score` DECIMAL(5,2) NOT NULL COMMENT '匹配得分(0-100)',
    `status` VARCHAR(32) DEFAULT 'PENDING' COMMENT '匹配状态',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_user_pair` (`user_id1`, `user_id2`),
    FOREIGN KEY (`user_id1`) REFERENCES `user`(`id`),
    FOREIGN KEY (`user_id2`) REFERENCES `user`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='匹配记录表';
```

## 8. 匹配算法

### 8.1 算法概述

匹配算法基于 **多维度加权打分** 机制，计算两个用户之间的相似度得分。

### 8.2 打分维度

| 维度 | 子项 | 单项分值 | 上限 |
|------|------|----------|------|
| **目标维度** | goals 交集 | 15 分/个 | 45 分 |
| **兴趣爱好** | hobbies 交集 | 10 分/个 | 35 分 |
| **学术维度** | GPA 区间一致 | 10 分 | 10 分 |
| **学术维度** | 专业一致 | 10 分 | 10 分 |
| **总计** | - | - | **100 分** |

### 8.3 算法流程

```
┌─────────────────────────────────────────────────────────────┐
│                    匹配算法流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 获取当前用户画像                                        │
│         ↓                                                   │
│  2. 查询其他所有用户画像                                    │
│         ↓                                                   │
│  3. 对每个候选用户:                                         │
│     - 计算 goals 交集 → 加分                                │
│     - 计算 hobbies 交集 → 加分                              │
│     - 比较 gpa_level → 加分                                 │
│     - 比较 major → 加分                                     │
│         ↓                                                   │
│  4. 过滤得分 > 0 的用户                                     │
│         ↓                                                   │
│  5. 按得分降序排序                                          │
│         ↓                                                   │
│  6. 取前 3 名                                                │
│         ↓                                                   │
│  7. 检查是否已存在匹配记录                                  │
│         ↓                                                   │
│  8. 创建新的 match_record                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.4 打分示例

```
用户 A:
  goals: ["准备直博", "科研组队", "MCM 参赛"]
  hobbies: ["吉他", "篮球", "摄影"]
  major: "计算机科学与技术"
  gpa_level: "Top 10%"

用户 B:
  goals: ["准备直博", "科研组队", "实习"]
  hobbies: ["吉他", "游泳"]
  major: "计算机科学与技术"
  gpa_level: "Top 20%"

计算:
  - goals 交集: ["准备直博", "科研组队"] → 2 × 15 = 30 分
  - hobbies 交集: ["吉他"] → 1 × 10 = 10 分
  - major 一致 → 10 分
  - gpa_level 不一致 → 0 分

总分: 30 + 10 + 10 + 0 = 50 分
```

## 9. 安全与权限

### 9.1 认证机制

**JWT Token 结构**:
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
{
  "sub": "student@life.hkbu.edu.hk",
  "userId": 1,
  "iat": 1711700000,
  "exp": 1711786400
}
```

**Token 管理**:
- 有效期：24 小时
- 存储：前端 localStorage
- 传输：Authorization Header

### 9.2 密码安全

**BCrypt 加密**:
- 自动盐值生成
- 成本因子：10
- 不可逆加密

### 9.3 接口权限

| 接口类型 | 权限要求 |
|---------|---------|
| 注册/登录 | 公开 |
| 用户数据 | 仅本人 |
| 画像更新 | 仅本人 |
| 匹配操作 | 仅本人 |

### 9.4 安全措施

- ✅ 校园邮箱白名单验证
- ✅ BCrypt 密码加密
- ✅ JWT Token 认证
- ✅ API 访问控制
- ⏳ 请求频率限制 (待实现)
- ⏳ 敏感操作日志 (待实现)
